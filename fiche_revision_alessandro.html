1. Architecture Django

Le cycle d'une requête HTTP

Navigateur  →  urls.py  →  views.py  →  models.py  →  BDD (SQLite)
                                    ↘  templates/  →  HTML  →  Navigateur

urls.py

Associe une URL à une vue.
path('films/', views.film_list)

views.py

Logique de la page : lit la BDD, valide le formulaire, retourne le HTML.

models.py

Définit les tables de la BDD sous forme de classes Python.

forms.py

Valide et nettoie les données envoyées par l'utilisateur (formulaires HTML).

templates/

Fichiers HTML avec la syntaxe Django ({{ var }}, {% tag %}).

Structure du projet banque_films

banque_films/          # Projet (configuration)

2. Les Modèles (models.py)

Principe

Un modèle = une classe Python = une table SQL. Chaque attribut = une colonne.

Types de champs essentiels

ChampÉquivalent SQLOptions utiles
CharField(max_length=100)VARCHAR(100)blank=True
TextField()TEXT (longueur libre)blank=True
IntegerField()INTEGERnull=True
DateTimeField()DATETIMEauto_now_add=True
ImageField(upload_to='...')VARCHAR (chemin fichier)null=True, blank=True
ForeignKey(Model, on_delete=...)Clé étrangère (N:1)related_name=
ManyToManyField(Model, through=...)Table pivot (N:N)through='Casting'

Options importantes

null vs blank

null=True → la colonne SQL peut être NULL
blank=True → le champ est optionnel dans les formulaires
Pour les champs texte : mettre les deux

on_delete (ForeignKey)

CASCADE → supprime aussi les objets liés
SET_NULL → met la clé à NULL
PROTECT → empêche la suppression

Exemple complet — modèle Film

class Film(models.Model):
    titre       = models.CharField(max_length=200)
    annee_sortie= models.IntegerField(null=True, blank=True)
    realisateur = models.CharField(max_length=100, blank=True)
    categorie   = models.ForeignKey(Categorie, on_delete=models.SET_NULL,
                      null=True, related_name='films')
    acteurs     = models.ManyToManyField(Acteur, through='Casting')

    class Meta:
        ordering = ['-annee_sortie', 'titre']  # tri par défaut

    def __str__(self):
        return self.titre  # affiché dans l'admin et les templates

Modèle utilisateur personnalisé

class Personne(AbstractUser):  # hérite de User Django
    TYPE_CHOICES = [('professionnel', 'Professionnel'), ('amateur', 'Amateur')]
    type = models.CharField(max_length=20, choices=TYPE_CHOICES, default='amateur')

# Dans settings.py : AUTH_USER_MODEL = 'films.Personne'
⚠️ AUTH_USER_MODEL doit être défini avant la première migration. Ne peut plus changer après.

3. Les Vues (views.py)

Structure d'une vue

def film_list(request):            # request = objet HTTP entrant
    films = Film.objects.all()    # requête BDD
    return render(request, 'films/films/list.html', {'films': films})
    #              template                              contexte (variables)

Fonctions raccourcis essentielles

FonctionUsage
render(request, template, contexte)Compile le template HTML et retourne la réponse
redirect('nom_url')Redirige vers une autre page (code HTTP 302)
get_object_or_404(Model, pk=pk)Récupère un objet ou retourne une erreur 404

Gérer GET et POST dans une même vue

def film_create(request):
    form = FilmForm(request.POST or None, request.FILES or None)
    # GET → request.POST est vide → form vide affiché
    # POST → request.POST contient les données → validation

    if request.method == 'POST' and form.is_valid():
        film = form.save()
        messages.success(request, 'Film ajouté !')
        return redirect('film_detail', pk=film.pk)

    return render(request, 'films/films/form.html', {'form': form})

Décorateur @login_required

from django.contrib.auth.decorators import login_required

@login_required          # si non connecté → redirige vers /login/
def film_create(request):
    ...

# Dans settings.py :
LOGIN_URL = '/login/'   # URL de redirection si non connecté

L'ORM Django — Requêtes courantes

CodeSQL équivalent
Film.objects.all()SELECT * FROM films_film
Film.objects.filter(categorie_id=3)WHERE categorie_id = 3
Film.objects.filter(titre__icontains='inc')WHERE titre LIKE '%inc%'
Film.objects.get(pk=5)WHERE id = 5 (lève exception si absent)
Film.objects.count()SELECT COUNT(*)
Film.objects.order_by('-annee_sortie')ORDER BY annee_sortie DESC
Film.objects.select_related('categorie')JOIN optimisé (évite N+1 requêtes)
Film.objects.get_or_create(titre='Inception')INSERT si absent, SELECT sinon

Requêtes avec Q() — Filtre OU

from django.db.models import Q

films = Film.objects.filter(
    Q(titre__icontains=q) | Q(realisateur__icontains=q)
)
# WHERE titre LIKE '%q%' OR realisateur LIKE '%q%'

4. Les Formulaires (forms.py)

ModelForm vs Form

ModelForm (lié à un modèle)

class FilmForm(forms.ModelForm):
    class Meta:
        model = Film
        fields = ['titre', 'annee_sortie']
Génère les champs automatiquement depuis le modèle. .save() sauvegarde en BDD.

Form simple (sans modèle)

class ConnexionForm(forms.Form):
    username = forms.CharField(...)
    password = forms.CharField(...)
Champs définis manuellement. Utilisé pour la connexion, la recherche, etc.

Méthodes importantes d'un formulaire

MéthodeDescription
form.is_valid()Valide les données, retourne True ou False
form.cleaned_data['champ']Données validées et nettoyées
form.save()Sauvegarde en BDD (ModelForm uniquement)
form.save(commit=False)Crée l'objet Python sans sauvegarder
form.errorsDictionnaire des erreurs de validation

Exemple — save(commit=False)

form = CommentaireForm(request.POST)
if form.is_valid():
    c = form.save(commit=False)  # objet Python créé, pas encore en BDD
    c.film = film                  # on ajoute le film manuellement
    c.personne = request.user      # on ajoute l'utilisateur connecté
    c.save()                       # maintenant on sauvegarde

6. Authentification

Connexion / Déconnexion

from django.contrib.auth import login, logout, authenticate

# Vérifier identifiants
user = authenticate(request, username='...', password='...')
# retourne l'objet User si OK, None si mauvais mdp

if user:
    login(request, user)   # crée la session (l'utilisateur est "connecté")

logout(request)            # supprime la session

Dans les templates

{% if user.is_authenticated %}
    Bonjour {{ user.username }} !
    <!-- user.is_staff → est-il admin ? -->
    <!-- user.type → professionnel ou amateur (notre champ perso) -->
{% else %}
    <a href="{% url 'login' %}">Se connecter</a>
{% endif %}

CSRF — Protection des formulaires

<form method="post">
    {% csrf_token %}   <!-- OBLIGATOIRE dans tout formulaire POST -->
    ...
</form>
⚠️ Sans {% csrf_token %}, Django renvoie une erreur 403 (Forbidden).

Résoudre un conflit

Quand deux personnes modifient le même fichier :

=======
leur code
>>>>>>> origin/main   # la version du serveur

→ Supprimer les marqueurs, garder le bon code, puis git add . et git commit.

Rôle de NGINX (serveur de Mathis)

Navigateur  →  NGINX (port 80/443)
                  ├─ /static/  →  fichiers CSS/JS (directement)
                  ├─ /media/   →  fichiers uploadés (directement)
                  └─ /...      →  Gunicorn → Django

NGINX est le "portier" qui distribue les requêtes. Django ne gère que la logique (pas les fichiers statiques en prod).

12. Questions type présentation

Q : C'est quoi un modèle Django ?

Une classe Python qui représente une table dans la base de données. Chaque attribut = une colonne. Django génère le SQL automatiquement via les migrations.

Q : Différence GET et POST ?

GET : lire des données (afficher une page). Les paramètres sont dans l'URL.
POST : envoyer des données (formulaire, connexion). Les données sont dans le corps de la requête.

Q : C'est quoi une migration ?

Un fichier Python qui décrit les changements à appliquer à la structure de la base de données. makemigrations le crée, migrate l'applique.

Q : Pourquoi @login_required ?

Pour protéger les vues qui modifient des données. Si l'utilisateur n'est pas connecté, il est redirigé vers /login/ automatiquement.

Q : C'est quoi le CSRF ?

Cross-Site Request Forgery : une attaque où un site malveillant envoie un formulaire à notre place. {% csrf_token %} ajoute un token unique qui prouve que le formulaire vient bien de notre site.

Q : ForeignKey vs ManyToMany ?

ForeignKey : N films → 1 catégorie.
ManyToMany : N films ↔ N acteurs (avec table pivot Casting).

Q : Pourquoi git pull avant de coder ?

Pour récupérer les modifications de l'équipe et éviter les conflits. Si on ne pull pas, on risque de travailler sur une version obsolète et d'avoir des conflits au push.

Q : DEBUG=False en production, pourquoi ?

En production, les pages d'erreur détaillées exposent le code source et les variables. DEBUG=False affiche une page générique à l'utilisateur et protège les informations sensibles.
